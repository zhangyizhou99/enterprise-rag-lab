# Enterprise RAG Lab 开发计划

## 目标

在 6 至 8 周业余时间内，完成一个可本地部署、可演示、可评测的企业文档 RAG 知识库。核心闭环是：

```text
脏文档接入 -> 可回放清洗 -> 混合检索 -> 可信问答 -> 评测 -> 反馈闭环
```

项目以真实工程过程为重点：用公开文档、真实评测数据和真实指标证明效果，不伪造企业数据或指标。

## 技术路线

| 模块 | 第一版选择 | 后续生产化替换 |
|---|---|---|
| 后端 | FastAPI | 保持不变 |
| 演示界面 | Gradio | React / Next.js |
| 元数据与反馈 | SQLite | PostgreSQL |
| 原始文件 | 本地目录 | MinIO / S3 |
| 向量检索 | Qdrant embedded | Qdrant Server / Milvus |
| Embedding | sentence-transformers 本地模型 | 私有部署或云 API |
| 关键词检索 | SQLite FTS5 或 rank-bm25 | Elasticsearch / OpenSearch |
| 生成模型 | OpenAI-compatible LLM API | 正式云 API 或自建模型 |
| 部署 | 本机启动 | Docker Compose |
| 指标 | SQLite 日志 + API | Prometheus / Grafana |

第一阶段不需要 Docker 或付费 API；先用本地 embedding、Qdrant embedded 和 SQLite 跑通闭环。

## Agent 定位与边界

本项目最终包含一个**知识库运营 Agent**，但它不是用 LLM 替代解析、清洗或检索的“万能聊天机器人”。

- **确定性服务负责事实**：文件格式识别、DOC 转换、解析、清洗规则、分块、索引、RRF、指标计算和数据写入均由可测试的普通代码完成。
- **Agent 负责决策与编排**：理解用户任务，选择并调用受控工具，解释检索证据、分析坏例、提出优化方案，并在授权后发起入库或评测任务。
- **默认只读**：Agent 可以查询语料、检索快照、评测结果和反馈；上传、重新入库、修改清洗规则、导出数据等写操作必须生成预览计划并等待用户确认。
- **每一步可审计**：保存用户目标、工具调用、参数、输入输出摘要、引用证据、模型与提示词版本、执行耗时和最终结论。Agent 没有证据时必须说明不确定性，而不是编造原因。

```text
用户目标
	-> Agent: 计划 -> 工具调用 -> 观察 -> 反思
	-> 受控服务: ingest / retrieve / evaluate / feedback analytics
	-> 证据、执行记录、预览计划
	-> 用户确认（仅写操作）
```

### Agent 工具表

| 工具 | 作用 | 默认权限 |
|---|---|---|
| `search_documents` | 检索文档和 chunk，返回来源、页码、格式和分数 | 只读 |
| `inspect_document` | 查看原文、清洗版本、规则命中和来源元数据 | 只读 |
| `inspect_run` | 查看入库、转换或索引任务的状态与错误 | 只读 |
| `evaluate_retrieval` | 对标注问题集计算 Recall@K、MRR、引用正确率 | 只读 |
| `analyze_feedback` | 按原因、文档、查询类型和索引版本分析点踩案例 | 只读 |
| `propose_cleaning_rule` | 根据坏例生成候选清洗规则与影响范围预览 | 只提案 |
| `plan_reingestion` | 计算受影响文档、预计 chunk 数和索引变更 | 只提案 |
| `run_ingestion` | 执行解析、清洗和索引任务 | 需用户确认 |
| `apply_cleaning_rule` | 写入并启用清洗规则 | 需用户确认 |

Agent 的首个可演示任务是：**“分析这条点踩答案为什么失败，并给出可验证的修复计划。”** 它必须区分清洗丢失、召回缺失、重排错误、证据不足或生成偏离，并链接当时的检索快照；没有证据则返回“无法确定”。

## 阶段 0：需求与数据集

**预计时间：1 至 2 天**

- 选择一个公开技术文档域，例如 FastAPI、Kubernetes、Python、LangChain 或开源项目官方文档。
- 收集 100 至 300 份 Markdown、PDF、DOCX 文档。
- 手工编写 30 条初始问题，并标注正确来源文档。
- 覆盖精确配置、故障排查、概念解释、步骤流程、跨文档比较等问题类型。
- 记录数据来源、许可证和文档版本。

**验收标准**：文档可追溯，初始评测集可运行。

**简历证据**：构建公开技术文档评测集，覆盖配置、排障、概念与流程类问题。

## 阶段 1：文档接入与清洗

**预计时间：1 周**

- 支持 Markdown、PDF、DOCX 的统一解析接口。
- 支持旧版 `.doc` 文件的受控转换：检测格式后用 LibreOffice `soffice` 转为 DOCX，保留原始哈希、转换器版本和中间产物映射；转换器缺失时输出可操作错误，不静默跳过。
- 定义 `Document`、`DocumentVersion`、`Chunk`、`IngestionRun` 数据模型。
- 保存原始文件、提取文本、清洗文本、清洗规则版本、解析失败原因。
- 清洗页眉页脚、页码、导航残留、重复段落、空白块、下载提示与无意义链接。
- 每次清洗生成统计：字符数变化、删除段落数、规则命中次数。
- 用随机抽样和回归测试防止新规则误删正文。
- [x] 对原生文本 PDF 识别表格边界与逻辑行，从重复横向起点恢复列，并用相邻页、重复表头、列起点及页边缘位置确定性合并跨页续表；RFC 9112 Table 2 已恢复为第 35 至 36 页的 3 列 8 行逻辑表。
- [ ] 扫描表格、复杂并排表格和视觉模型兜底暂不实现；后续必须独立评测，不能复用原生文本 PDF 的结论。

**验收标准**：任意文档均可查看“原始文本 -> 清洗后文本 -> 规则命中记录”。

**简历证据**：构建可回放文档清洗流水线，保留原始文件、清洗版本与规则命中记录，并通过抽样回归测试控制误删风险。

## 阶段 2：分块、索引与基础检索

**预计时间：1 周**

- [x] 基于标题层级、段落和代码块进行结构化分块；普通文本优先使用行、句子和单词边界，超长代码优先在当前窗口后半段寻找换行，否则按字符上限兜底。
- [ ] 将已识别的超长表格按完整行分块、重复表头，并将逐页 bbox 与 `page_end` 传播到最终 Chunk；当前表格仍保持原子性。
- [x] 每个 chunk 保存文档 ID、标题路径、页码、chunk 序号、前后相邻 chunk。
- [x] 批量处理 125 份 Markdown，并隔离、汇总单文件失败。
- [x] 修复全量语料中的仅标题 Chunk 和超长代码 Chunk，并重建最新 BM25 索引。
- [x] 实现可注入 Embedding 编码器、稳定版本身份、`float32` 持久化、token 截断审计和调试 CLI。
- [x] 安装 `sentence-transformers`，使用固定模型 commit 为 127 份文档的最新 Chunk 生成真实 embedding，并验证维度、范数与持久化数量。
- [x] 根据全库 E5 token 审计将普通文本上限从 1,200 校准到 900 字符，以 chunker 0.4.0 重建 1,362 个最新 Chunk；最大 token 数由 572 降至 488，最新 Embedding 截断数由 6 降至 0，并增加默认拒绝截断的存储前防线。
- [x] 用 Qdrant embedded 建立绑定模型 commit 和全部 Embedding 成员的不可变向量快照，完成 `query: ` 编码、余弦 Top-K、来源输出和幂等同步。
- [x] 实现 SQLite FTS5 trigram 与 BM25 关键词检索。
- [x] 实现检索调试 CLI，展示来源、分数和命中片段。
- [x] 实现版本化评测数据契约、当前 Chunk/证据/快照校验、Hit Rate@K、Recall@K、MRR、延迟统计和逐题候选报告。
- [x] 生成首批 30 条种子问题，配置、概念、流程、排障和比较类各 6 条，并生成分批人工审核材料。
- [x] 由项目作者逐条确认问题、相关度、证据和遗漏候选；q013–q014、q019 按反馈补充证据后通过，最终 `approved = 30/30`、42 条 Chunk judgment 均通过正式门禁。

**验收标准**：对初始 30 条问题，可展示 Top-K 召回的真实 chunk 及出处。

当前基础链路已能展示 BM25 与向量 Top-K 的真实 Chunk 及出处，并能生成版本化评测报告。BM25 整句精确短语导致 30 题全空的故障已修复为 `bm25_fts5_trigram_or_v0.2.0`；30 题全部人工批准后的正式 Top-5 基线为 BM25 Hit Rate 0.800、Recall 0.711、MRR 0.661，E5/Qdrant Hit Rate 0.867、Recall 0.767、MRR 0.692。阶段 2 验收完成，阶段 3 已进入 RRF 实现与消融。

**简历证据**：设计包含标题路径、页码和相邻块关系的 Chunk 元数据模型，为可追溯检索与上下文扩展提供依据。

## 阶段 3：混合检索、父子扩展与 Rerank

**预计时间：1 至 2 周**

- [ ] 在 BM25、向量检索和首批标注集稳定后，再实现版本化 Query 预处理；支持规范化、phrase/all/any 模式和可解释的降级记录，不在当前基础检索阶段展开。
- [x] 将 BM25 Top-20 与向量 Top-20 按 `chunk_id` 去重，使用固定 $k=60$ 的 RRF 融合并输出 Top-5；保存两路 rank/score、RRF score 和稳定 tie-break，不混合异构原始分数。
- [x] 命中 Chunk 后按需补充前后邻居：只接受同标题、标题父子或同父兄弟关系，PDF 不跨页；每个锚点使用独立字符预算，并跨锚点去重。
- [ ] 加入可替换 reranker 接口，初期可先关闭以完成消融实验。
- [ ] 增加语义结果校验：记录 query-chunk embedding 余弦相似度并与人工相关性标签对照；相似度只作为诊断特征，不单独判定结果正确。
- [x] 保存 RRF 与上下文检索快照：query、候选、两路排名与分数、融合分数、向量快照 ID、扩展关系/距离和最终上下文。
- [ ] 对比混合检索 + rerank；BM25、向量、纯 RRF、RRF + 章节相邻扩展的消融已完成。

当前固定参数 Top-5 消融结果：BM25 为 Hit Rate 0.800、Recall 0.711、MRR 0.661；E5/Qdrant 为 0.867、0.767、0.692；RRF 为 0.833、0.733、0.672。RRF 没有超过向量基线，主要坏例是两路中等排名的重叠候选压过仅一路高排名的候选。RRF + depth=1 章节相邻扩展不改变锚点指标，扩展证据 Hit Rate/Recall/MRR 为 0.900/0.883/0.696；代价是平均每题新增 5.77 个 Chunk，且 95.4% 扩展项未被当前 judgment 覆盖。下一步实现可插拔 reranker，以同一评测集检验能否在保持证据覆盖的同时压缩上下文。

**验收标准**：输出至少包含 Recall@5、MRR 与 P95 检索延迟的真实评测表。

**当前简历证据**：实现 BM25 与向量召回的确定性 RRF 融合及带章节/页边界、字符预算和跨锚点去重的相邻上下文扩展，保留各阶段分数与来源快照，并在 30 条人工批准问题上完成四路消融；可插拔 rerank 尚未实现。

## 阶段 4：带引用问答与拒答机制

**预计时间：1 周**

- 提供 `/ask` API：检索 -> 证据拼装 -> LLM 回答 -> 返回引用。
- 约束模型只能根据给定证据回答；关键结论必须带 chunk 引用。
- 使用 Top-1 分数、有效证据数量、rerank 分数和引用覆盖率设计生成前拒答门槛。
- 不满足门槛时不调用 LLM，或仅返回“未检索到可靠依据”及候选资料。
- 保存模型、检索配置、提示词版本与完整回答快照。

**验收标准**：在评测集上区分正确回答、合理拒答与错误回答，并统计三者比例。

**简历证据**：设计基于证据分数、有效引用数量和重排置信度的生成前拒答策略，降低无依据回答风险。

## 阶段 5：演示界面、反馈与分析

**预计时间：1 周**

- 用 Gradio 完成上传、文档浏览、聊天、引用展开界面。
- 每条回答保存唯一 `answer_id`。
- 支持点赞、点踩和原因：检索无关、答案错误、引用不准确、信息不完整。
- 反馈入库时保存检索快照，而不只保存赞/踩结果。
- 增加反馈分析页：点踩率、原因分布、最常出问题的文档、最近坏例。
- 支持把点踩样本导入待标注评测集。
- 只有在积累足够的 query-chunk 分组相关性标签后，才把 LambdaMART 作为可选 Learning-to-Rank 实验；输入可包含 BM25、向量相似度、RRF 排名、标题命中、Chunk 长度和来源特征，并与无训练的 RRF及 reranker 基线做消融比较。

**验收标准**：能完整演示“提问 -> 显示引用 -> 点踩 -> 查看检索快照 -> 进入待评测队列”。

**简历证据**：实现答案级反馈闭环，持久化问题、候选片段、阶段分数与配置版本，使坏例可定位至清洗、召回、重排或生成环节。

## 阶段 6：知识库运营 Agent

**预计时间：1 周**

- 实现 AgentCore：`plan -> act -> reflect`，并为工具定义输入输出 schema。
- 接入 `search_documents`、`inspect_document`、`inspect_run`、`evaluate_retrieval`、`analyze_feedback` 五个只读工具。
- 为工具调用记录 run ID、参数、结果摘要、证据引用、模型/提示词版本和耗时。
- 首先只支持“分析点踩案例”和“解释某个检索结果为何被召回”两类任务。
- 为 `propose_cleaning_rule` 和 `plan_reingestion` 增加只提案预览，不允许 Agent 直接改数据。
- 为所有写工具实现确认令牌：没有用户确认则拒绝执行。
- 建立 Agent 评测用例：工具选择正确率、证据引用率、无证据拒答率和写操作确认覆盖率。

**验收标准**：对至少 10 个真实或构造的坏例，Agent 能引用检索快照定位失败环节；无证据时不归因；未确认的写操作 100% 被拦截。

**简历证据**：实现具备 Plan-Act-Reflect 编排、工具 schema、证据溯源和人审门的知识库运营 Agent，可基于检索快照分析点踩坏例并生成可验证的修复计划。
